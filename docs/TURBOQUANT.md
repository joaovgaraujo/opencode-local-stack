# TurboQuant (experimental)

TurboQuant (Google Research, ICLR 2026) applies a random Hadamard rotation
before scalar quantization, giving noticeably better quality-per-bit than
standard GGUF quants at very low bit widths — appealing for a 35B MoE model
you're trying to squeeze onto an 8 GB card.

**It is not part of official llama.cpp.** Support exists only as independent
community forks, none merged upstream as of when this doc was written. That
matters for two reasons:

1. **Format**: standard llama.cpp builds cannot load a TurboQuant GGUF at all —
   they don't recognize the quant type.
2. **Binaries**: a few forks publish prebuilt CUDA binaries, but they're
   compiled by individual community members, not the ggml-org project.
   `install.py` deliberately **does not download or execute** any of them —
   downloading and silently running a third-party compiled binary with GPU
   driver access is a real supply-chain risk, and picking one fork over
   another isn't a call this installer should make for you.

## What `install.py` does for you

If you pick the TurboQuant entry for Qwen3.6-35B-A3B, it downloads the actual
TQ GGUF weights (verified real file, from
[`mad-lab-ai/Qwen3.6-35B-A3B-tq-gguf`](https://huggingface.co/mad-lab-ai/Qwen3.6-35B-A3B-tq-gguf)),
prints this same warning, and stops there. Everything past that is manual.

## To actually test it

1. Build (or fetch, if you trust the source and have reviewed it) a
   TurboQuant-enabled llama.cpp. A few forks with prebuilt releases, as of
   this writing:
   - https://github.com/atomicmilkshake/llama-cpp-turboquant (Windows CUDA 13, prebuilt binaries on Hugging Face)
   - https://github.com/TheTom/llama-cpp-turboquant (Mac Metal + Windows CUDA 12.4, builds on tag push)
   - https://github.com/AtomicBot-ai/atomic-llama-cpp-turboquant (Windows CUDA 13.3, DLLs bundled)
   - https://github.com/spiritbuun/buun-llama-cpp (CUDA, validated on an NVIDIA GPU)

   **Review before running.** These are third-party binaries; at minimum,
   check the repo's build workflow (GitHub Actions log) matches what it
   claims to build, and prefer building from source yourself if you can.

2. Point the installer or a run script at that build instead of the
   downloaded release:
   ```
   python install.py --model qwen3.6-35b-a3b \
       --quant qwen3.6-35b-a3b-instruct-TQ3_1S.gguf \
       --bin-dir /path/to/your/turboquant-llama.cpp \
       --non-interactive
   ```
   `install.py` will skip the llama.cpp download (binary already found in
   `--bin-dir`) and just download the weights + start the server with the
   normal flags. **Add the fork's TurboQuant flags yourself** — they differ
   per fork, typically something like:
   ```
   --cache-type-k turbo3 --cache-type-v turbo3    # or turbo4, or -tq3/-tq4 — check the fork's README
   ```
   `run.ps1` / `run.sh` (regenerated by the installer) won't include these —
   edit them in by hand after the first run, or pass extra args directly to
   `llama-server`.

3. Compare against the standard Q4_K_M quant using `tests/validate.py` (point
   `LLAMA_BASE_URL`/`LLAMA_MODEL` at whichever server is running) and
   `tests/vram_logger.ps1` / a `nvidia-smi --query-gpu=memory.used -l 2` loop
   on Linux.

If you find a verified TurboQuant GGUF repo for one of the other five models
in the catalog, adding it is a one-entry change in `installer/catalog.py` —
just verify the exact filename/size against the HF API first (see
[`MODELS.md`](MODELS.md)).
